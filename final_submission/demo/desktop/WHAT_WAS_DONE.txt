1. DATABASE SCHEMA (PostgreSQL)
------------------------------------------

Created 10 new database tables that work together:

  ORGANIZATION STRUCTURE:
  - organizations: Company/firm accounts
  - departments: Divisions within companies (Accounting, Finance, etc.)
  - teams: Smaller groups within departments
  - roles: Job titles with permission levels (Owner, Manager, Employee, etc.)
  - employees: Staff members linked to roles and departments

  FINANCIAL DATA:
  - transactions: All financial records (money moving in/out)
  - documents: Bank statements, receipts, invoices
  - reconciliations: Matching transactions to documents

  AUDIT & COMPLIANCE:
  - audit_logs: Complete history of who did what, when
  - variances: Flagged suspicious transactions for review
  - permissions: Role-based access control rules

Why this structure?
- Multi-tenant: Each organization is completely isolated (legally required)
- Audit-ready: Every change is tracked with timestamps and user IDs
- Scalable: Uses UUID keys instead of sequential IDs (better for distributed systems)
- Linked: Transactions can be connected to documents and audit logs
- Flexible: JSONB columns allow storing complex data about changes


2. RUST BACKEND SERVICES (The Business Logic)
------------------------------------------

Created three service modules that handle database operations:

  services/organizations.rs
  - Create and retrieve organizations
  - Manage departments, teams, roles
  - Methods are:
    * create_organization(owner_id, name, type)
    * create_department(org_id, name, description)
    * create_team(dept_id, org_id, name, description)
    * create_role(org_id, name, description, permission_level)
    Plus all the "get" methods to retrieve data

  services/transactions.rs
  - Create financial transactions
  - Flag suspicious transactions as "variances"
  - Resolve flagged variances
  - Methods include:
    * create_transaction(org_id, account_name, amount, date)
    * flag_variance(org_id, tx_id, variance_type, description)
    * resolve_variance(variance_id, resolved_by)

  services/audit.rs
  - Log every action (who did what, when)
  - Retrieve audit trails for any entity
  - Retrieve user action history
  - Methods include:
    * log_action(org_id, user_id, entity_type, entity_id, action)
    * get_entity_audit_trail(org_id, entity_type, entity_id)
    * get_user_actions(org_id, user_id, limit)

Why Rust?
- Type safety: Compiler catches errors at build time, not runtime
- Performance: Rust is extremely fast
- Memory safety: No null pointer errors or buffer overflows
- Matches your existing Tauri architecture

HOW IT ALL WORKS TOGETHER
================================================================================

When someone uses Balancd:

1. They log in (existing auth system)
2. They go to CompanyStructure page
3. Page calls: await api.organizations.get(orgId)
4. This calls Tauri command: get_organization
5. Tauri invokes Rust function: OrganizationService::get_organization()
6. Rust queries database: SELECT * FROM organizations WHERE id = ?
7. Database returns organization data
8. Rust returns to Tauri command
9. Tauri command returns to JavaScript
10. API service receives and logs
11. React component displays the organization
12. User sees their company name, type, creation date

When they create a new department:

1. User fills form: name="Accounting", description="..."
2. User clicks "Add Department"
3. Form calls: await api.organizations.createDepartment(...)
4. Tauri command: create_department
5. Rust function: OrganizationService::create_department()
6. Rust generates new UUID and current timestamp
7. Database INSERT: INSERT INTO departments VALUES (...)
8. Database creates audit log: INSERT INTO audit_logs VALUES (...)
9. Returns new department record
10. React component adds to list
11. User sees new department appear immediately

When they record a transaction:

1. User enters: account="Bank", amount=5000, date="2026-03-12"
2. Calls: await api.transactions.create(...)
3. Database creates transaction with created_by=current_user
4. Audit log created automatically
5. Component displays in table
6. If AI later detects anomaly, flag_variance can be called
7. Variance appears in "Variances" tab
8. User can review and resolve it
9. Another audit log entry is created

All of this is TRACKED in audit_logs, including:
- Who created it
- When they created it
- What they created
- Any changes they made later
- When those changes were made
- By whom

TECHNICAL DECISIONS & WHY
================================================================================

DECISION: Why PostgreSQL specifically?
REASON: You discussed "cloud database storage" for larger enterprises and
        PostgreSQL is free, reliable, and handles JSON data well via JSONB.
        It supports multi-tenancy patterns easily.

DECISION: Why Rust for backend services?
REASON: Matches your Tauri choice. Type safety. Performance. You already have
        Tauri setup, so Rust is natural.

DECISION: Why UUID for primary keys?
REASON: Better for distributed systems and data replication. Sequential IDs
        can cause issues with horizontal scaling. UUIDs prevent collisions.

DECISION: Why log changes to JSONB instead of separate tables?
REASON: Maximum flexibility. Audit logs need to store arbitrary changes from
        different entity types. JSONB handles this cleanly.

DECISION: Why array columns for linked_document_ids?
REASON: Simple relationships. A transaction can link to multiple documents
        (bank statement, receipt, invoice). Arrays are simpler than junction
        tables in this case.

DECISION: Why three separate components not one mega-component?
REASON: Separation of concerns. Each has one job:
        - CompanyStructure: org management
        - TransactionsManagement: financial records
        - AuditTrail: compliance/history
        Easier to test, maintain, and extend separately.

DECISION: Why create_file instead of using ORM like Diesel?
REASON: Direct SQL queries are simpler for this scope. ORMs add complexity
        when you don't need their power. Keep it readable.

UPDATES - MARCH 12 EVENING SESSION (BUG FIXES)
================================================================================

After initial infrastructure work, testing revealed state management issues.
A series of Tauri compiler errors occurred when attempting to login:

  ERROR: "state not managed for field `logger`"
  ERROR: "called `Result::unwrap()` on an `Err` value: ColumnDecode { index: 
          "id", source: "mismatched types; Rust type `i32` is not compatible 
          with SQL type `UUID`" }"

ROOT CAUSE ANALYSIS & RESOLUTION
----------------------------------

Issue 1: Multiple Generic State<T> Parameters
Problem: Tauri's #[tauri::command] macro doesn't properly handle multiple 
         State<T> parameters, especially with Option<T>. The macro expansion 
         becomes ambiguous.

Solution: Created unified AppState struct combining all managed states
         - New file: src-tauri/src/state.rs
         - Single State parameter instead of dual parameters
         - All 24 command handlers refactored to use AppState

Issue 2: Type Mismatches (UUID vs i32 vs String)
Problem: Database uses UUID for all IDs, but Rust UserAccount struct used i32
         Then changed to String, but database still expected UUID type.

Solution: Updated UserAccount and Organization types to use uuid::Uuid
         - Uuid type has built-in serde support
         - Matches database UUID column type directly
         - No manual string conversion needed

Issue 3: Duplicate Module Declarations
Problem: main.rs declared its own "logging" module, creating two distinct
         SumoLogger types - one local, one from lib.rs in AppState

Solution: Removed local mod logging from main.rs
         Import SumoLogger directly from lib: use desktop_lib::logging::SumoLogger

ARCHITECTURE DECISION: Unified State Pattern
----------------------------------

This refactoring establishes the correct pattern for Tauri apps with multiple
managed state values:

CORRECT APPROACH (what we now have):
  pub struct AppState {
      pub db_pool: Option<PgPool>,
      pub logger: SumoLogger,
  }
  
  pub async fn command(state: State<'_, AppState>) { ... }

AVOID (what caused the original error):
  pub async fn command(
      db: State<'_, Option<PgPool>>,
      logger: State<'_, SumoLogger>
  ) { ... }

This pattern is now documented in code and should be followed for any future
additions to managed state.
